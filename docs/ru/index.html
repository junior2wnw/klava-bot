Локальный desktop-агент. Явные подтверждения. Реальная работа на машине.

Чините рабочие станции, проводите миграции и меняйте локальную машину без потери следа.

Klava нужен для работы, которая не должна исчезать в истории чата: восстановление рабочей станции, рискованные локальные изменения, guarded terminal execution, миграции репозитория и провайдера, а также операторские циклы, где важны receipts.

Почему этот репозиторий вообще стоит открыть

Он пытается быть полезнее, чем просто “AI chat with tools”.

Klava держит локальную работу в одном accountable loop: проверить машину, запросить approval, выполнить ограниченное действие, сохранить transcript, оставить recovery hints и artifacts, которые потом сможет проверить другой инженер.

Для чего это

Stateful local workflows

Работа, где в одном цикле встречаются машина, терминал, approvals, проверки и rollback context.

Для чего это не нужно

Не только ради vibe coding

Klava ориентирован на recoveries, migrations и operator-facing workflows, где receipts важнее эффектных chat demos.

Почему это можно продолжать

Durable memory и journal

Context compression, semantic retrieval и restart-safe execution journal снижают шанс потерять нить или начать блефовать.

Representative workflows

Примеры сценариев, где Klava может реально окупаться

Ремонт рабочей станции

Проверить устройства, подготовить rollback context, провести guarded fix и оставить результат в task history.

Миграция репозитория

Переписать конфигурацию, обновить адаптеры, прогнать smoke checks и оставить bounded change record.

Смена AI-провайдера

Перевести проект между провайдерами, не разбрасывая model lists, validation steps и rollback hints по чатам.

Внутренние IT-процессы

Держать рискованные действия за явными approvals, а не прятать их в free-form privileged prompting.

Consultant runbooks

Заменить ad-hoc notes и shell history на один task log с командами, approvals и machine facts.

Support handoff

Выгружать support bundles и transcript так, чтобы другой инженер мог продолжить кейс без реконструкции с нуля.

Начинать лучше с evidence

Сначала релиз и код, потом уже разговор о пилотах.

Если вы оцениваете проект для команды, начинайте с релиза, runtime proof и safety docs. Коммерческий путь вторичен по отношению к продуктовым доказательствам.
