# Support

Use the right channel for the right kind of problem.

## Bugs

Open a GitHub issue if you have:

- a reproducible defect
- a UI/runtime failure
- packaging problems
- provider integration bugs

Please include:

- operating system
- Klava version
- exact steps
- expected vs actual result
- logs or screenshots when possible

## Feature requests

Open a GitHub issue or discussion if you have:

- a concrete workflow Klava should support
- an architectural improvement
- a product direction suggestion tied to a real user problem

## Security issues

Do not open a public issue first for a sensitive vulnerability.

See [`SECURITY.md`](./SECURITY.md).

## Questions

Use GitHub Discussions for:

- setup help
- architecture questions
- product direction conversations
- integration and extension ideas

## What makes support fast

The fastest reports are the ones that include:

- exact version
- exact platform
- exact reproduction
- logs with timestamps
- whether the issue is local, provider-side, or packaging-related

Klava is designed to produce support bundles for exactly this reason.
